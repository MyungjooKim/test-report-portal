# Test Result Portal — 결과 취합 기능 설계 문서 (개발 착수용)

> 대상: TR Portal (`tr_ui`, Node.js/Express + Vanilla JS)
> 목적: 파편화된 테스트 결과서(매뉴얼·자동화, 다양한 포맷)를 **TC ID 기준으로 취합**해 실시간 대시보드·pivot으로 보여준다.
> 이 문서는 그대로 구현에 착수할 수 있도록 순서대로 작성되었다. 시각 자료는 §13 참조.

---

## 1. 목표와 범위

- 매뉴얼 QA와 자동화 팀이 각각 기록한 결과서를 **하나로 취합**해서 본다.
- 취합 결과물은 **물리 통합 문서가 아니라** 소스에서 실시간 계산되는 **대시보드 + pivot 표**. (실시간 업데이트 대응)
- 필요할 때만 **통합본을 내보내기(export)**로 생성(시점 고정 산출물).
- 입력 포맷은 Google Sheets에 한정하지 않고 **CSV·Excel·MD·HTML** 등도 수용.
- 결과 취합이 아닌 **일반 문서 열람** 용도도 공존해야 하므로 **폴더 유형**으로 성격을 분리한다.
- 비범위(후속): 중앙 TC 관리(TC Manager), 스냅샷 기반 실행 이력 일원화. 본 기능은 그 징검다리.

## 2. 현재 상태 (As-Is)

현재 TR Portal 동작:
- 좌측 **스페이스(폴더 트리)**로 프로젝트를 관리. (이 구조는 유지)
- 프로젝트 선택 → `리포트 업로드` → **파일 1개 = 리포트 1건**으로 목록에 추가(1:1).
- 리포트 클릭 → 개별 열람(HTML/MD 렌더, Google Sheets 연동). 통계는 단건 기준.
- 여러 파일을 올려도 서로 **합쳐지지 않음**. 크로스 소스(매뉴얼 vs 자동화) 비교 불가.

관찰된 실제 데이터 구조 (샘플: `PARAMETA_Supercycl_TestCase_v3.1`):
- **TC ID**: `SC-TRD-FTPS-034`, `SC-COMM-001` 형태. 접두사+번호, 고유. (병합 키)
- **스냅샷/버전**: 문서명의 `v3.1`.
- **결과 컬럼**: 단일 컬럼이 아니라 담당자/환경별 다중 컬럼(`Tester 1`, `Tester 2`, `Win/MAC`, `MAC` 등).
- **결과값**: `Pass`, `Fail`, `N/T`(미수행), `N/A`.
- **거래소 차원**: `HL`(Hyperliquid), `BG`(Bitget), `GT`(Gate) 등. 마스터 TC가 거래소별로 확장됨.
- 기타 컬럼: 카테고리(대/중/소분류), 사전 조건, 스텝, 기대 결과, 우선순위, 실제 결과·N/A 사유·특이사항.
- 현재 파일엔 **자동화 전용 컬럼은 없음**(이전 형식).

## 4. 폴더/프로젝트 유형

프로젝트 생성 시 **유형을 선택**한다. 유형이 업로드 방식과 화면을 결정한다.

| 유형 | 업로드 동작 | 메인 화면 | 트리 아이콘 |
|------|-------------|-----------|-------------|
| 결과형 (테스트 결과) | 취합 옵션 모달(§6) | 통합 대시보드 + pivot 표(§9.3) | 📊 |
| 문서형 (일반 문서/자료) | 단순 파일 업로드(현행) | 파일 목록 + 뷰어(md/html/이미지) | 📄 |

정책 (확정): **폴더 유형 고정 (A)**.
- 폴더 성격은 **생성 시 정해지고 이후 고정**. 결과형 폴더 = 취합·대시보드 단위, 문서형 폴더 = 뷰어.
- **결과형 폴더에 올라온 비결과 파일**(TC ID·결과 표가 없는 md·이미지·메모 등)은 **취합에서 제외**하고, 해당 폴더의 **"첨부/기타" 영역에 일반 파일(뷰어)로 표시**한다. → 폴더 성격은 유지하면서 참고 파일은 부드럽게 수용.
- 결과 파일 판별: **TC ID + 결과가 담긴 표(구조화) 감지 시 취합 대상**, 아니면 첨부로 분류.
- (B안 "파일 개별 인식"은 오인식·동작 예측 저하·집계 범위 모호·구현 비용 때문에 채택하지 않음.)

> 스키마: `projects` 테이블에 `type ENUM('result','doc')` 컬럼 추가. 미지정(레거시)은 `doc`로 취급해 기존 동작 보존.

## 5. 결과 취합 — 사용자 플로우

1. **프로젝트 생성** — 유형 = 결과형 선택.
2. **결과 파일 업로드** — 1개 또는 여러 개.
3. **자동 감지** — 파일 수/컬럼 구성으로 취합 방식 제안(§6).
4. **매핑 확인** — TC ID 컬럼, 스냅샷, (필요 시) 소스/결과 컬럼 지정.
5. **미리보기** — 매칭 N · 미매칭 X · 소스별 결과 수 · 불일치 수. 문제 있으면 4로.
6. **취합 실행** — 정규화(long) 후 병합.
7. **통합 결과 열람** — 실시간 대시보드 + pivot 표.
8. **(선택) 내보내기** — 시점 고정 통합본(xlsx/Google Sheet).

> 두 취합 방식(§6)은 4~5단계 이후 **동일 파이프라인으로 수렴**한다. 방식과 무관하게 결과물은 같다.

## 6. 취합 방식 옵션 (소스 레이아웃)

"어떻게 합칠까"가 아니라 **"내 결과가 어디에 있나"**를 고르는 옵션. 내부 병합 로직은 동일.

1. **파일 분리형 (여러 파일)** — 매뉴얼·자동화가 각각 별도 파일. 업로드 시 **파일별 소스 태그** 지정.
2. **단일 파일형 (한 파일·다중 컬럼)** — 한 파일에 매뉴얼·자동화 결과 컬럼이 함께. **컬럼 → 소스 매핑**.

안전장치:
- **자동 감지 → 사용자 확인**: 자동화 컬럼/파일 수를 감지해 방식을 제안하고, 사용자는 확인·수정만.
- **미리보기 필수**: 커밋 전 매칭/불일치 통계로 오설정을 잡는다.

### 6.1 옵션 층위 명확화 (2026-07-21 추가 — 필수 고려)

실제 사용 시나리오에서 사용자가 가장 먼저 고르는 것은 **결과 파일의 양식(포맷/도구)** 이다. 팀·도구마다 결과서 양식이 다르기 때문에, 업로드 시 옵션을 제공해 선택하게 하고 그 뒤 각 양식의 데이터 소스로 대시보드를 구성한다. 따라서 업로드 옵션은 다음 3층위로 구성한다:

1. **양식/소스 유형 선택** — Playwright 리포트 / Excel·CSV / Google Sheets / MD·HTML 등. 어댑터 `supports()` 자동 감지 + 사용자 확인.
2. **소스 레이아웃** — 파일 분리형 / 단일 파일형 (본 절의 기존 옵션 2종).
3. **키 매핑** — TC ID 컬럼 / 스냅샷 / 결과값 표준화.

구현 원칙: **특정 양식 전용의 별도 업로드 경로를 만들지 않는다.** 모든 양식(Playwright 포함)이 이 공통 옵션 플로우(§5의 3~5단계: 자동 감지 → 매핑 확인 → 미리보기)로 진입한 뒤 동일 파이프라인으로 수렴해야 한다.

## 7. 다포맷 수용 — 어댑터 계층

"포맷을 읽는 계층(어댑터)"과 "합치는 계층(정규화·병합)"을 분리한다. 어댑터는 어떤 포맷이든 공통 정규화 레코드(§8.1)를 반환.

| 포맷 | 파서 | 비고 |
|------|------|------|
| Google Sheets | Sheets API (기존 OAuth 토큰 재사용) | 실시간 재조회에 유리 |
| Excel `.xlsx` | SheetJS 등 | 시트 다중 탭 지원 |
| CSV | csv 파서 | 단순 |
| Markdown | `marked`/remark로 **GFM 표** 파싱 | 표가 있어야 취합 가능 |
| HTML 리포트 | `cheerio`로 `<table>` 파싱 | 자동화 리포트 표 대응 |

공통 어댑터 인터페이스(제안):
```ts
interface ResultAdapter {
  supports(file): boolean;
  parse(file, opts): {
    rows: Array<{ tcId: string; resultRaw: string; cols: Record<string,string> }>;
    detected: { tcIdColumn?: string; resultColumns?: string[]; snapshot?: string };
  };
}
```

전제(수용 조건):
1. 각 파일은 **TC ID + 결과가 표(구조화)로** 존재해야 자동 취합됨. 서술형만 있는 md/html은 취합 대상이 아니라 **문서형(뷰어)**으로 처리.
2. **결과값 표기 통일** 필요(§8.3). 파일마다 `Pass/PASS/성공/O/✓` 등 상이 → 표준값으로 매핑.

새 포맷 지원 = **어댑터 하나 추가**로 확장.

## 8. 데이터 모델

### 8.1 정규화 레코드 (long)

모든 소스를 이 형태로 저장/계산한다. (한 행 = TC × 소스 × (환경/거래소) 하나의 결과)

| 필드 | 타입 | 설명 |
|------|------|------|
| project_id | string | 소속 프로젝트(폴더) |
| snapshot | string | 예: `v3.1` |
| tc_id | string | `SC-...` (병합 키) |
| source | enum | `manual` / `automation` (확장: tester 명) |
| exchange | string? | `HL`/`BG`/`GT`/`공통` 등 (없으면 null) |
| env | string? | `Win/MAC` 등 (없으면 null) |
| result | enum | `Pass`/`Fail`/`N/T`/`N/A`/`Blocked` |
| reason_note | text? | N/A 사유·특이사항 |
| source_file_id | string | 출처 파일 |
| imported_at | datetime | 취합 시각 |

### 8.2 소스 파일 레지스트리

| 필드 | 설명 |
|------|------|
| id | 파일 식별자 |
| project_id | 소속 프로젝트 |
| filename / url | 파일명 또는 Google Sheet URL/ID |
| format | gsheet/xlsx/csv/md/html |
| source_role | manual/automation |
| snapshot | 감지/지정값 |
| mapping_json | tcIdColumn, resultColumns→source, valueMap 등 |
| last_synced_at | 마지막 재조회 시각 |

> 소스 파일을 **등록**해두고 앱이 당겨와 계산한다. 사용자가 24개를 수동 병합하는 게 아니라 **한 번 등록 → 시스템이 재조회·재계산**.

### 8.3 결과값 표준화 맵 (설정값)

```
Pass    ← pass, PASS, 성공, O, ✓, OK
Fail    ← fail, FAIL, 실패, X
N/T     ← N/T, NT, 미수행, 미실행, (빈칸)
N/A     ← N/A, NA, 해당없음, 제외
Blocked ← blocked, 보류, 차단
```
미인식 토큰은 **`미인식`으로 표시 + 매핑 요청**(미리보기에서 노출).

### 8.4 병합·pivot 규칙

- 그룹 키: `tc_id` (+ 필요 시 `exchange`).
- 소스별 결과를 컬럼으로 pivot: `매뉴얼`, `자동화`, `최종`.
- **최종 상태 규칙(기본 제안, 설정 가능)**:
  - 둘 다 있고 값이 같으면 → 그 값.
  - 값이 다르면 → **불일치(mismatch)** 플래그, 최종은 보수적으로 `Fail`(정책 선택).
  - 하나만 있으면 → 그 값.
  - `N/T`/`N/A`는 비교 대상에서 제외(커버리지·불일치 계산 시).
- **불일치**: 매뉴얼 result ≠ 자동화 result (둘 다 Pass/Fail일 때).
- **자동화 커버리지**: 자동화 결과가 있는 TC 비율(%).
- **미매칭**: 결과엔 있으나 TC 마스터(또는 다른 소스)에 없는 TC.

## 9. 화면 사양

### 9.1 새 프로젝트 모달
- 필드: 이름, **유형(결과형/문서형)**, 비공개 toggle.
- 유형 선택 = 이후 업로드/메인 화면 분기 결정.

### 9.2 업로드 / 취합 옵션 모달 (결과형)
- 취합 방식 카드 2개(§6), 자동 감지 추천 배지.
- 파일 분리형: 파일 목록 + 파일별 소스 드롭다운.
- 단일 파일형: 컬럼 → 소스 매핑 UI.
- 키 매핑 확인: TC ID 컬럼 / 스냅샷 / 결과값 표준화.
- **미리보기 통계**: 전체 TC · 매뉴얼 · 자동화 · 미매칭 · 불일치.
- 액션: 취소 / 취합 실행.
- 목업: `qa-consolidation-upload-mockup.html`.

### 9.3 통합 결과 대시보드 (결과형 메인)

> **(2026-07-21 개정)** 기존 단건 리포트 대시보드의 검증된 구성(게이지 + 축별 스택 바 + Fail 분석)을 재사용한다. 숫자 카드 나열 + 원시 pivot 표만으로는 부족(러프)하다는 피드백 반영. 화면 순서 = 정보 우선순위.

- **① 요약 밴드** (기존 `renderDashboardPanel` 문법 재사용):
  - Pass Rate **원형 게이지** + 실행률 게이지 (모수 관계 표시: 전체 = 실행(P+F) + N/T, N/A 제외).
  - Pass / Fail / N/T / Total 카운트.
  - 불일치·소스별 커버리지 카드는 **소스 2종(매뉴얼+자동화) 이상일 때만 노출** — 단일 소스에서 "불일치 0"은 무의미.
- **② 축별 분포** (기존 `renderStackRow` 스택 바 재사용): 거래소별 / 영역(suite·분류)별 / 소스별 Pass·Fail·N/T 누적 바. 바 클릭 = 하단 pivot 필터 적용.
- **③ Fail 인사이트**: 실패 **사유 패턴 그룹핑 Top N**(예: Timeout, toBeVisible, toContainText…) + 건수, 클릭 시 해당 행 필터. 기존 AI Fail 분석(결함 후보) 연계.
- **④ 소스 파일**: **접힌 요약 한 줄**(소스 N개 · 역할 · 마지막 동기화 시각) + 펼치면 목록·관리(삭제/원본 열기/새로고침). 인사이트 위에 두지 않는다.
- **⑤ pivot 표** (상세 탐색): `TC ID · 매뉴얼 · 자동화 · 최종`(+ 거래소/환경 축), 필터(전체/Fail/불일치/N/T)+검색, 불일치 하이라이트. **고정 높이 스크롤 박스** 안에 표시(단건 대시보드의 Fail 목록과 동일한 방식) — 표가 페이지를 끝없이 늘리지 않는다.
- **⑥ AI** (기존 단건 대시보드와 동일 컴포넌트 재사용):
  - **Fail 분석 및 결함 후보** — 자동 생성(지문 캐시), 서버가 축별 분포·사유 패턴을 결정적으로 집계하고 AI는 클러스터링·해석만.
  - **AI Q&A** — 통합 pivot을 컨텍스트로 질문. 수량 질문은 `define_metric` 도구로 계산(어림 금지).
- **첨부/기타 영역**: 결과 파일이 아닌 파일(취합 제외)을 일반 뷰어로 열람. (§4 정책)
- `결과 소스 업로드`, `통합본 내보내기` 버튼.

### 9.4 문서형 메인 (현행 유지)
- 파일 목록 + md/html/이미지 뷰어. `파일 업로드`(단순).

## 11. 엣지 케이스 · 검증 규칙

- TC ID 컬럼 감지 실패/모호 → 매핑 단계 강제.
- 한 소스 내 TC ID 중복 → 플래그(최신 우선 or 오류 선택).
- 스냅샷 불일치(파일 간 버전 상이) → 경고.
- 결과값 미인식 토큰 → `미인식` 표시 + 매핑 요청.
- 거래소/환경 다차원: 동일 TC ID가 거래소별로 존재 → 키에 `exchange` 포함하거나 집계 방식 선택.
- 대용량(시트 24개+): 소스별 파싱을 배치/비동기로, 진행률 표시.
- 표 없는 md/html·이미지가 결과형에 업로드됨 → **취합 제외 + "첨부/기타" 영역에 일반 파일로 표시**(§4 확정 정책). 오분류로 취합됐다면 사용자가 "첨부로 이동" 가능.

## 12. 구현 순서 (단계별)

- **Phase 0 — 유형 분리**: `projects.type` 추가, 유형별 라우팅. 문서형=현행 유지, 결과형=빈 대시보드 스켈레톤. (회귀 없음 확인)
- **Phase 1 — 소스 수집 + 어댑터(우선 CSV/XLSX/GSheet)**: 업로드/등록, 파싱→정규화 저장.
- **Phase 2 — 매핑 + 자동 감지 + 미리보기**: TC ID/결과 컬럼 감지, 통계 미리보기.
- **Phase 3 — 병합 + 통합 대시보드/pivot(실시간)**: 최종 상태·불일치·커버리지, 새로고침.
- **Phase 4 — MD/HTML 어댑터** 추가.
- **Phase 5 — 내보내기**(xlsx/gsheet).
- **Phase 6 — 인사이트 고도화**: 거래소/환경 축 pivot, 불일치 상세, 추세.

## 13. 수용 기준 (핵심)

1. 결과형 프로젝트에 매뉴얼 파일 + 자동화 파일(다른 포맷 포함)을 올리면 TC ID로 병합돼 하나의 pivot 표로 보인다.
2. 방식(분리형/단일형)과 무관하게 동일한 통합 결과가 나온다.
3. 소스가 갱신되면 새로고침으로 최신 반영되고, 물리 통합 문서를 수동 관리하지 않는다.
4. 매뉴얼 Pass / 자동화 Fail 같은 **불일치가 표시**된다.
5. 문서형 프로젝트는 기존처럼 파일 목록 + 뷰어로 동작(회귀 없음).
6. 통합본 내보내기로 시점 고정 xlsx/Sheet를 생성할 수 있다.

## 부록 — 에이전트용 TL;DR
- 폴더에 `type`(result/doc) 도입 → 유형별 화면·업로드 분기. 문서형=현행.
- 결과형: 여러 포맷(gsheet/xlsx/csv/md/html) 결과 파일을 **어댑터로 파싱 → TC ID·스냅샷·소스·결과로 정규화(long) → 병합**.
- 취합 옵션(분리형/단일형)은 소스 레이아웃 선택일 뿐, 병합 결과는 동일. 자동 감지 + 미리보기로 보조.
- 통합본은 유지 대상이 아니라 **실시간 계산 대시보드/pivot**, 내보내기는 on-demand.
- 병합 키 = TC ID(+snapshot, 필요 시 exchange). 결과값은 표준화 맵으로 통일. 불일치·커버리지 산출.
