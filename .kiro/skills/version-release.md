# Version Release Skill

사용자가 "버전 올려줘", "version up", "version release" 등의 문구를 사용하면 이 스킬을 실행한다.

## 프로세스

### 1. 변경 내용 분석
- 마지막 git tag 이후의 커밋 메시지를 수집한다 (`git log <last-tag>..HEAD --oneline`)
- 태그가 없으면 전체 커밋 히스토리를 확인한다
- 변경 내용을 분류한다:
  - **Added**: 새 기능
  - **Changed**: 기존 기능 수정/개선
  - **Fixed**: 버그 수정
  - **Removed**: 제거된 기능

### 2. 버전 결정 (Semantic Versioning)
- 커밋 내용을 분석하여 자동으로 버전 레벨을 결정한다:
  - **patch** (0.0.X): 버그 수정, 텍스트 수정, 소소한 개선
  - **minor** (0.X.0): 새 기능 추가, UI 개선
  - **major** (X.0.0): 호환이 깨지는 변경, 구조 변경, API/DB 스키마 변경
- 항상 0.0.0 (3자리)부터 시작한다

### 3. 사용자 확인
- 다음 내용을 사용자에게 보여주고 최종 확인을 받는다:
  - 현재 버전 → 새 버전
  - 변경 내용 요약 (릴리즈 노트 미리보기)
  - 버전 레벨 (patch/minor/major) 및 결정 이유
- 사용자가 승인하면 진행, 수정 요청 시 반영 후 재확인

### 4. 버전 갱신 실행
- 프로젝트 내에서 버전이 기록된 모든 위치를 찾아서 갱신한다:
  - `package.json`의 `version` 필드
  - HTML/코드 내 버전 표기 텍스트 (예: `v0.0.1`)
  - 기타 설정 파일에 버전이 명시된 곳
- 검색 방법: `grep -r` 등으로 현재 버전 문자열이 있는 파일을 탐색

### 5. CHANGELOG.md 작성
- 프로젝트 루트에 `CHANGELOG.md`가 없으면 새로 생성한다
- 파일 상단에 새 버전 항목을 추가한다 (최신이 위로, 누적 방식)
- 형식:

```markdown
## [X.Y.Z] - YYYY-MM-DD

### Added
- 간단명료한 설명

### Changed
- 간단명료한 설명

### Fixed
- 간단명료한 설명
```

- 내용은 사람이 읽고 바로 이해할 수 있는 가장 간단한 수준으로 작성
- 해당 카테고리에 내용이 없으면 해당 섹션은 생략

### 6. Git 커밋 & 태그
- 변경된 파일을 커밋한다: `chore: release vX.Y.Z`
- 태그를 생성한다: `git tag vX.Y.Z`
- push 한다: `git push && git push --tags`

## 주의사항
- 버전 갱신 전에 반드시 사용자 확인을 받는다
- 작업 중인 변경사항(uncommitted changes)이 있으면 먼저 커밋을 권유한다
- 릴리즈 노트는 한글 또는 영어 중 프로젝트에서 사용 중인 언어를 따른다
